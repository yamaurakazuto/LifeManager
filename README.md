# LifeManager

## Overview
LifeManager は、日々の収支を管理・集計するためのデスクトップアプリケーションです。WPF を用いてシンプルなUIを構築し、ユーザーが日付ごとの収入・支出を確認できるように設計しています。

本プロジェクトは単なる機能実装ではなく、「設計」、特に依存関係の整理を意識したアプリケーション構築を目的としています。WPF を通して、UIとロジックの分離や責務の明確化を実践しています。

---

## Architecture

本アプリケーションは MVVM パターンをベースに、以下のレイヤー構成を採用しています。

View → ViewModel → UseCase → DTO

各レイヤーの責務は以下の通りです。

- View  
  UIの表示のみを担当します。ロジックは持たず、ViewModelにバインドすることで状態を反映します。

- ViewModel  
  UIの状態を管理します。ユーザーの操作を受け取り、UseCaseを呼び出し、その結果をUIに反映します。

- UseCase  
  ビジネスロジックを担当します。データの取得や集計処理を行い、ViewModelに返します。

- DTO  
  データの受け渡し専用のオブジェクトです。レイヤー間の依存を弱め、構造を明確にする役割を持ちます。

---

## Data Flow

ユーザー操作からUI更新までの流れは以下のようになります。

User Action  
↓  
ViewModel  
↓  
UseCase  
↓  
DTO  
↓  
ViewModel  
↓  
UI  

ViewModelはUseCaseから受け取った結果をUI用の状態として保持し、バインディングを通してViewに反映します。

---

## Design Intent

WPFは単なるUIフレームワークではなく、依存関係の設計がそのままアプリケーションの品質に直結するフレームワークだと考えています。

そのため本プロジェクトでは以下の点を重視しています。

1. ビジネスロジックの分離  
計算処理やデータ操作はすべてUseCaseに集約し、ViewModelには持ち込まないようにしています。これにより責務を明確にし、変更しやすい構造にしています。

2. 依存関係の明確化  
ViewはViewModelのみを参照し、ViewModelはUseCaseに依存する一方向の構造にしています。これにより構造がシンプルになり、影響範囲が限定されます。

3. データ構造の分離  
DTOを導入することで、UIとビジネスロジック間のデータ構造を明確に分離しています。これによりレイヤー間の結合を弱く保ちます。

4. 状態とロジックの分離  
ViewModelは状態管理に専念し、ロジックはUseCaseに委譲することで、それぞれの責務を明確にしています。

## Thoughts

本プロジェクトを通して、WPFはUIを作るための技術というよりも、設計力を問われるフレームワークであると感じました。

特に依存関係が曖昧なまま開発を進めると、構造がすぐに崩れてしまう一方で、責務を明確に分離することで、非常に保守性の高い設計が可能になります。

そのため本プロジェクトではUseCase層を導入し、ロジックと状態を分離することで、将来的な拡張や変更に耐えられる構造を意識しました。
