## 1. このプロジェクトは何か

**LifeManager** は、日々の収支（収入・支出）を記録する Windows デスクトップアプリ。

| 項目 | 内容 |
|------|------|
| 種類 | WPF アプリ（Web アプリではない。Windows 上で動く GUI アプリ） |
| フレームワーク | .NET 8（`net8.0-windows`） |
| UI ライブラリ | Material Design In XAML Themes 5.3.1（ボタンやアイコンの見た目を提供） |
| DB | SQL Server（接続部分は実装途中。現在はダミーデータで動く） |
| 設計パターン | MVVM（次章で説明） |

### 開発環境の準備と実行方法
1. Visual Studio 2022（「.NET デスクトップ開発」ワークロード入り）をインストール。
2. リポジトリをクローンし、`LifeManager.sln` を開く。
3. **F5 キー（デバッグ実行）** でアプリが起動する。
   - コマンドラインなら `dotnet build LifeManager.csproj` でビルドできる。
   - SQL Server は無くても起動する（DB 接続は一部画面の暫定コードのみ）。

## 2. 前提知識：MVVM とは（このプロジェクトの読み方）

WPF アプリでよく使われる設計パターン。クラスを3つの役割に分けることで、
「見た目」と「ロジック」を分離し、UI なしでロジックをテストできるようにする。

```
┌──────────┐  Binding   ┌────────────┐  呼び出し  ┌──────────────┐
│  View     │ ◄────────► │ ViewModel   │ ─────────► │ UseCase / DB │
│ (XAML)    │            │ (状態+操作) │            │ (業務ロジック) │
└──────────┘            └────────────┘            └──────────────┘
   見た目だけ              画面の状態を保持           計算・データ取得
```

| 用語 | 意味 | このプロジェクトでの実物 |
|------|------|--------------------------|
| **View** | 画面の見た目。XAML ファイルとそのコードビハインド（.xaml.cs） | `Views/` フォルダ |
| **ViewModel** | 画面に表示する状態（数値・一覧）と操作（コマンド）を持つクラス | `ViewModel/` フォルダ |
| **Model** | DB の1行に対応するデータの入れ物 | `Models/` フォルダ |
| **Binding** | XAML の `{Binding Xxx}` 構文。View の表示と ViewModel のプロパティを自動同期する仕組み | 各 .xaml 内 |
| **Command** | ボタン押下などの操作を ViewModel のメソッドに繋ぐ仕組み（`ICommand`） | `Commands/RelayCommand.cs` |
| **DTO** | 層と層の間でデータを受け渡すための入れ物クラス | `Application/DTOs/` |
| **UseCase** | 「日次サマリーを取得する」のような業務上の1操作を表すクラス | `Application/UseCases/` |
| **Repository** | DB アクセスを閉じ込める層（SQL をここ以外に書かない） | `Repositories/`（実装予定） |

**大原則：ViewModel は View（Window）を直接知らない。**
画面遷移すら `INavigationService` という抽象（インターフェイス）越しに依頼する。
この「依存の向き」を理解すると、コード全体が読めるようになる。

## 6. 作業2：UIリデザイン案モックの追加（PR #3 / commits `2db017c` → `ffb4002`）

### 何をしたか
デザイン確認用のモック画面（本番とは独立した Window）を追加した。

```
追加  Views/Mockup/UiRedesignMockWindow.xaml       … モック画面本体
追加  Views/Mockup/UiRedesignMockWindow.xaml.cs    … コードビハインド（InitializeComponent のみ）
削除  docs/mockups/intuitive-ui.html               … 最初に作った HTML 版（後述の経緯で削除）
```

### なぜしたか
「既存のダークな配色（背景 `#0F172A`、カード `#1E293B`、Teal / Violet のアクセント）を
活かして、化粧パレットのように *見た瞬間に何をどこで操作するか分かる* UI にしたい」
という要望に対するデザイン提案。実画面をいきなり書き換えると差し戻しコストが高いため、
まず独立したモックで方向性を合意する進め方をとった。

### デザインの方針
- **収入＝ティール（`#2DD4BF`）／支出＝ローズ（`#FB7185`）** で役割を色分け
  → ラベルを読まなくても色で判別できる
- **合計は符号で色が変わる**（プラス＝ティール、マイナス＝ローズ）
- **操作の縦導線を1本化**：日付 → 収入 → 支出 → 合計 → 保存 を上から順に配置
- **ホームは大きな2択タイル**（「収支をつける」「履歴をみる」）で最初の一歩を迷わせない
- 主要アクション（保存）はバイオレット（`#8B5CF6`）

### 経緯（なぜ HTML 版が一度作られて消えたのか）
1. 最初、ブラウザで手軽に確認できる HTML でモックを作成した（commit `2db017c`）。
2. しかし本アプリは WPF 製。HTML では実際の描画（Material Design テーマ、WPF の
   レイアウト挙動）を再現できず忠実性が低い、という指摘を受けた。
3. そこで実テーマで描画される WPF/XAML モックへ作り直し、HTML を削除した
   （commit `ffb4002`）。**現在有効なのは XAML 版のみ。**

### モックの見方・読み方
- **見る**: Visual Studio で `Views/Mockup/UiRedesignMockWindow.xaml` を開くと、
  デザイナ（プレビュー画面）にアプリを起動せずに表示される。
- **読む**: XAML を上から読むと「ヘッダ → ホーム用 Border → 収支入力用 Border →
  収支履歴用 Border」の順で、そのまま画面の並びに対応している。
- 表示値（120,000 円など）はすべてハードコード。Binding は使っていない。
  これは「見た目の確認」だけが目的で、実データやロジックに影響を与えないため。

## 7. 作業3：デバッグ実行でモックを確認できるようにする（PR #4 / commit `5891a7f`）

### 何をしたか
F5（デバッグ実行）でモック画面がそのまま立ち上がるように、起動設定を変更した。

| ファイル | 変更 |
|----------|------|
| `App.xaml` | `StartupUri`（最初に開く画面の指定）を `Views/MainWindow.xaml` → `Views/Mockup/UiRedesignMockWindow.xaml` に変更 |
| `App.xaml.cs` | `OnStartup` 内の `new MainWindow {...}` をコメントアウト |

### なぜしたか
モックを追加しただけでは F5 しても表示されない。理由は2つ:
1. `StartupUri` が従来の `MainWindow` を指したままだった。
2. その `MainWindow` はコンストラクタで SQL Server への接続
   （`SqlConnection.Open()`）を試みるため、DB が無い環境では起動時に例外で落ちる。
   `App.xaml.cs` 内の `MainWindow` 生成コードは元々 Show されない未使用コードだったが、
   生成しただけでコンストラクタは走るので、同じ理由で例外の原因になる。

この2点を解消し、**SQL Server が無い環境でも F5 一発でモックを確認できる** ようにした。

### ⚠ これは暫定措置（本番に戻す手順）
デザイン確認が終わったら、以下の2手順で本番フローに戻す（`App.xaml.cs` の
コード内コメントにも同じ手順を明記してある）:
1. `App.xaml` の `StartupUri` を `Views/MainWindow.xaml` に戻す。
2. `App.xaml.cs` の `OnStartup` 内のコメントアウトを解除する。

### 動作確認済みの内容
- `dotnet build` 成功（0エラー）。
- ビルド済み exe を実際に起動し、SQL Server なしで例外なくモックが描画されることを確認。

## 8. コードの読み進め方（初めての人向け・推奨ルート）

作業1で書き直したコメントは「なぜこの設計か」を説明しているので、
以下の順にファイルを開いてコメントを読みながら進めると、全体像が最短で掴める。

### Step 1: 起動の入口（アプリはどこから始まるか）
1. `App.xaml` … `StartupUri` が最初に開く画面を決める。**現在はモックを指している**（作業3）。
2. `App.xaml.cs` … `OnStartup`。本来は Service → ViewModel → View の順に依存を
   組み立てる場所（Composition Root と呼ぶ）。現在は作業3のため中身を一時コメントアウト中。

### Step 2: メイン画面と画面遷移（MVVM の肝）
3. `Views/MainWindow.xaml` … ボタンが `Command="{Binding ...}"` で ViewModel に繋がる。
4. `Views/MainWindow.xaml.cs` … `DataContext` に `MainViewModel` を設定する。
   ※末尾に DB 接続の疎通確認コードが残っている（本来は Repository の責務・暫定コード）。
5. `ViewModel/MainViewModel.cs` … ボタン押下 → `INavigationService` に画面表示を「依頼」する。
6. `ViewModel/INavigationService.cs` … 画面遷移の契約（インターフェイス）。
7. `Serviecs/NavigationService.cs` … 契約の実装。**Window を new してよいのはここだけ**。

> ここまでで「ViewModel は View を知らない。抽象越しに依頼するだけ」という
> 依存の向き（依存性逆転）が理解できる。これがこのプロジェクトの設計の核。

### Step 3: 収支入力画面（状態管理と業務ロジックの分離）
8. `Views/TransactionsView.xaml` … 入力欄・一覧・合計・保存。Binding と入力検証の設定。
9. `Views/TransactionsView.xaml.cs` … 「数字しか入力させない」という UI 都合の処理だけ持つ。
10. `ViewModel/TransactionViewModel.cs` … 画面の状態を保持する中心クラス。
    - `INotifyPropertyChanged` … 値の変更を View に通知して再描画させる仕組み
    - `IDataErrorInfo` … 入力検証（収入は0以上 等）を ViewModel 側に集約する仕組み
    - `Total` … Income − Expense を常に計算で導出（二重管理を防ぐ）
11. `Application/UseCases/GetDailySummaryUseCase.cs` … 取得・集計ロジック。
    **現在はダミーデータを返す**。後で DB 取得に差し替えても ViewModel は無変更で済む設計。
12. `Application/DTOs/` … `TransactionDto` / `DailySummaryDto`。層をまたぐデータの形。

### Step 4: 履歴画面とデータ層（実装途中の部分）
13. `ViewModel/HistoryViewModel.cs` … `ObservableCollection` で一覧を保持。
14. `Views/HistoryWindow.xaml(.cs)` … コードビハインドは `InitializeComponent` のみ（意図的）。
15. `Models/DailySummaries.cs` … DB の1行＝1日分のスナップショットに対応するモデル。
16. `Repositories/DailySummaryRepository.cs` … DB アクセス層。**中身は未実装（クラス定義のみ）**。
17. `Commands/RelayCommand.cs` … ボタン操作を ViewModel のメソッドに変換する汎用部品。

### Step 5: モックを見る
- Visual Studio で `Views/Mockup/UiRedesignMockWindow.xaml` を開く（デザイナでプレビュー）、
- または `design/mock-debug-preview` ブランチに切り替えて F5。
